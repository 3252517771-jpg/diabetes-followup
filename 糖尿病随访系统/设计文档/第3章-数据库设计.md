# 第3章 数据库设计

## 3.1 概念结构设计

### 3.1.1 核心实体

#### 用户相关实体

| 实体 | 说明 | 主要属性 |
|------|------|---------|
| User | 系统用户，包含所有角色 | 用户ID、用户名、密码哈希、姓名、手机号、角色、科室、状态、创建时间 |
| Role | 角色定义 | 角色ID、角色名称、权限JSON |
| UserRole | 用户-角色关联 | 用户ID、角色ID |

#### 患者相关实体

| 实体 | 说明 | 主要属性 |
|------|------|---------|
| Patient | 糖尿病患者档案 | 患者ID、姓名、性别、年龄、手机号、诊断类型、严重程度、状态、标签列表、负责医生ID、创建时间、更新时间 |
| PatientTag | 患者标签/分组 | 标签ID、标签名称、颜色 |

#### 随访计划相关实体

| 实体 | 说明 | 主要属性 |
|------|------|---------|
| FollowupTemplate | 随访计划模板 | 模板ID、名称、描述、适用糖尿病类型、总时长、阶段数、创建者ID、是否公用、创建时间 |
| FollowupStage | 计划阶段 | 阶段ID、模板ID、阶段序号、阶段名称、开始偏移天数、持续天数 |
| StageTask | 阶段任务项 | 任务ID、阶段ID、任务类型（血糖测量/电话随访/到院复查/饮食记录）、执行方（患者/医护）、频率、提醒提前时间 |
| FollowupPlan | 计划实例（应用于患者） | 计划ID、模板ID、患者ID、创建医生ID、开始时间、预计结束时间、当前阶段、状态（进行中/已结束/已中止） |

#### 数据记录相关实体

| 实体 | 说明 | 主要属性 |
|------|------|---------|
| BloodGlucoseRecord | 血糖记录 | 记录ID、患者ID、测量值、测量时间、分类（空腹/餐后/睡前/随机）、血糖单位、异常标记、记录来源（患者录入/医护录入） |
| DietRecord | 饮食记录 | 记录ID、患者ID、餐次（早餐/午餐/晚餐/加餐）、食物名称、分量描述、记录时间 |
| DietRecommendation | 饮食推荐 | 推荐ID、患者ID、生成方式（AI/手动）、推荐内容JSON、审核状态（待审/已通过/已驳回）、审核人ID、推荐时间、推送状态 |
| PatientSelfReport | 患者自述记录 | 自述ID、患者ID、分类、内容、医生回复、回复时间、创建时间 |

#### 通知相关实体

| 实体 | 说明 | 主要属性 |
|------|------|---------|
| NotificationTemplate | 消息模板 | 模板ID、名称、触发事件、模板内容（含占位符）、渠道类型 |
| NotificationLog | 通知发送日志 | 日志ID、接收人（患者或用户）、关联计划ID、模板ID、发送渠道、发送内容、发送时间、状态（成功/失败）、失败原因 |

#### 系统相关实体

| 实体 | 说明 | 主要属性 |
|------|------|---------|
| SystemConfig | 系统全局配置 | 配置键、配置值、描述、更新时间 |
| OperationLog | 操作审计日志 | 日志ID、操作用户ID、操作类型、操作描述、操作对象类型、对象ID、IP地址、操作时间 |

### 3.1.2 实体关系（E-R 要点）

```
User ──1:N──> Patient                          # 一个医生负责多个患者
User ──N:N──> Role  (through UserRole)         # 用户可拥有多个角色
User ──1:N──> FollowupPlan                     # 一个医生创建多个计划实例
User ──1:N──> DietRecommendation               # 一个营养师审核多个推荐
User ──1:N──> OperationLog                     # 一个用户产生多个操作日志

Patient ──1:N──> FollowupPlan                  # 一个患者可有多个随访计划实例
Patient ──1:N──> BloodGlucoseRecord            # 一个患者有多条血糖记录
Patient ──1:N──> DietRecord                    # 一个患者有多条饮食记录
Patient ──1:N──> DietRecommendation            # 一个患者有多条饮食推荐
Patient ──1:N──> PatientSelfReport             # 一个患者有多条自述记录
Patient ──N:N──> PatientTag                    # 一个患者可有多个标签

FollowupTemplate ──1:N──> FollowupStage        # 一个模板有多个阶段
FollowupStage ──1:N──> StageTask               # 一个阶段有多个任务项
FollowupTemplate ──1:N──> FollowupPlan         # 一个模板产生多个计划实例
FollowupPlan ──1:N──> NotificationLog          # 一个计划产生多条通知记录

NotificationTemplate ──1:N──> NotificationLog  # 一个模板产生多条通知日志

## 3.3 物理结构设计

### 3.3.1 存储引擎

所有表统一使用 **InnoDB** 引擎，支持事务、外键约束和行级锁。

### 3.3.2 字符集

- 数据库字符集：`utf8mb4`
- 排序规则：`utf8mb4_unicode_ci`

### 3.3.3 索引设计

| 表名 | 索引字段 | 索引类型 | 说明 |
|------|---------|---------|------|
| sys_user | username | UNIQUE | 用户名唯一索引 |
| patient | responsible_doctor_id | INDEX | 按负责医生快速查询 |
| patient | status | INDEX | 按患者状态筛选 |
| patient | phone | INDEX | 按手机号搜索 |
| patient | name | INDEX | 按姓名搜索 |
| followup_plan | patient_id | INDEX | 按患者查计划 |
| followup_plan | creator_id | INDEX | 按创建医生查计划 |
| followup_plan | status | INDEX | 按计划状态筛选 |
| blood_glucose_record | (patient_id, measure_time) | COMPOSITE | 患者血糖按时间排序查询 |
| blood_glucose_record | (patient_id, category) | COMPOSITE | 患者按类型统计血糖 |
| diet_record | (patient_id, record_time) | COMPOSITE | 患者饮食记录按时间查询 |
| diet_recommendation | patient_id | INDEX | 按患者查推荐 |
| diet_recommendation | review_status | INDEX | 按审核状态筛选 |
| patient_self_report | (patient_id, created_at) | COMPOSITE | 按患者和时间查自述记录 |
| notification_log | (recipient_type, recipient_id) | COMPOSITE | 按接收方查通知 |
| notification_log | plan_id | INDEX | 按计划查通知 |
| notification_log | status | INDEX | 按发送状态筛选 |
| sys_operation_log | (user_id, created_at) | COMPOSITE | 按用户和时间查询操作日志 |
| sys_operation_log | target_type | INDEX | 按对象类型筛选 |
| sys_config | config_key | UNIQUE | 配置键唯一 |

### 3.3.4 分区策略

当前阶段不设置分区。当 `blood_glucose_record` 表数据量超过 500 万行时，考虑按月份进行 RANGE 分区。

### 3.3.5 备份策略

- 每日全量备份（凌晨 02:00）
- 保留最近 30 天备份
- 使用 `pg_dump` 进行逻辑备份
```
