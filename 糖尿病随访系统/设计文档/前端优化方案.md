# 糖尿病随访系统 · 前端优化方案

> 版本：v0.1（讨论定稿版）
> 状态：讨论中，持续更新
> 目标：在 **不影响原始功能、真实接口对接、权限逻辑与路由结构** 的前提下，对前端表现层进行细节优化与动效增强。

---

## 一、文档用途

本文件用于在前端优化讨论过程中持续沉淀已确认结论，避免多轮对话后丢失上下文。  
当前阶段 **仅讨论方案，不直接修改代码实现**。

---

## 二、总目标

前端优化阶段聚焦以下范围：

1. 提升全局框架层的视觉识别度
2. 引入 ReactBits 组件作为局部动效增强
3. 优化页面标题区、导航交互与关键视觉节奏
4. 保持医生端后台的可用性、稳定性与业务连续性

---

## 三、执行摘要

本轮前端优化遵循“**表现层升级，业务层稳定**”原则，重点不是重做产品，而是在既有真实业务链路之上提升识别度、层级感与审阅效率。

当前已形成的总体方向如下：

- 全局框架层以 `StaggeredMenu` 和 `TextType` 建立新的导航与标题语言
- 登录页以 `ProfileCard` 建立品牌化入口，但不改变登录流程
- 工作台与血糖模块借鉴 `Magic Bento` 的排版思路，强化图表与信息主次关系
- 患者管理重点优化患者详情页，而非高频患者列表
- 饮食推荐重点优化审核详情，通过 `ScrollStack` 建立分层审阅节奏
- 消息中心重点优化详情抽屉，提升消息判断与处理效率
- 随访计划、系统管理本轮不做视觉重构，以控制风险与范围

## 四、任务边界

### 3.1 可调整范围

- 布局表现形式
- 导航呈现方式
- 页面标题区视觉与动效
- 空状态、过渡态、局部交互动效
- ReactBits 组件引入方式与封装方案
- 页面局部排版、间距、层级、视觉节奏

### 3.2 禁止改动范围

- 前后端真实 API 对接方式
- 路由结构与路由路径
- 权限判断逻辑
- 菜单项的业务含义与跳转目标
- 表单提交流程
- 认证流程与退出登录逻辑
- 各业务模块的数据结构、字段语义、接口契约
- 已完成并可用的核心业务功能

### 3.2.1 本轮可接受的结构调整标准

- 本轮前端优化允许：
  - **少量结构调整**
- 但前提是：
  - 不影响功能可用性
  - 不影响真实接口对接
  - 不影响权限逻辑
  - 不影响核心业务流程
- 若某项优化需要大规模结构重组，应单独提出并重新确认，不默认纳入本轮范围

### 3.3 动效使用限制

- 强动效优先集中在导航展开、标题进入、空状态提示等低风险区域
- 表格、表单、图表主体区域不做高干扰动效
- 所有新增动效需具备可降级能力，不影响信息读取与操作效率

### 3.4 ReactBits 落地原则

- 对已确认接入的 ReactBits 组件，采用：
  - **尽量贴近原示例**
  - 但必须 **服从医疗后台 / 医疗工具场景气质**
- 落地时优先保留组件的：
  - 核心结构
  - 核心交互逻辑
  - 核心识别性
- 不追求对 ReactBits 示例进行完全自由改造
- 也不允许为了还原示例而破坏本项目的业务可用性与阅读效率

## 六、全局基线结论

### 5.1 全局框架层

#### 导航

- 现有左侧固定侧栏将被替换为 ReactBits 的 `StaggeredMenu`
- 菜单采用 **左侧滑出** 方式
- 菜单触发按钮位于 **顶栏左上角**
- 菜单整体气质偏 **ReactBits 展示版**
- 菜单允许更强的层叠、展开与视觉冲击感

#### 标题区

- 页面标题区将引入 ReactBits 的 `TextType`
- `TextType` 应用于：
  - 英文 kicker
  - 中文主标题
- 动效形式采用 **打字机式**
- 页面进入时逐字出现，完成后保持静止
- 不采用多句轮播切换

#### 保持不变

- 菜单项数量与权限逻辑不变
- 路由与跳转目标不变
- 顶栏用户信息保留
- 退出登录功能保留
- 内容区业务交互保持原样

#### 当前判断

- 全局框架层将成为本轮前端优化中视觉变化最明显的区域
- 强动效主要集中在菜单展开/收起与标题区，不扩散到核心业务操作区域

## 八、模块设计定稿记录

### 7.1 登录页

#### 已确认

- 登录页希望使用 ReactBits 的 `ProfileCard`
- `ProfileCard` 用作 **登录页视觉主卡片外壳**
- 登录页仍保留标准登录表单能力：
  - 用户名输入
  - 密码输入
  - 错误提示
  - 登录按钮
- 不改变现有登录流程、认证逻辑与登录成功后的跳转逻辑
- 登录表单将作为一层 **覆盖在 `ProfileCard` 前景上的浮动面板**
- `ProfileCard` 动效强度采用 **全开**
- 浮动表单面板采用 **轻实体信息板** 风格
- 默认测试账号提示继续保留，并明确展示：
  - `doctor01 / secret123`
  - `admin01 / secret123`

#### 当前判断

- 登录页优化重点是“视觉承载方式重做”，不是“交互范式重做”
- `ProfileCard` 应承担品牌感、氛围感与入口识别度
- 真正的登录操作仍应保持后台系统的可理解性与可操作性
- 即便保留 `ProfileCard` 的完整表现力，也不能影响：
  - 输入框聚焦
  - 错误提示可读性
  - 登录按钮点击
  - 鼠标与触控操作稳定性
- 登录页背景只引入轻量英文短句 / 医疗感标语，不使用大段说明文案

#### 待继续确认

- 无（当前模块已通过）

### 7.2 工作台看板

#### 已确认

- 工作台优化目标同时包含：
  - 信息编排优化
  - 动效表现优化
- 但优先级为：**先保证高效医疗后台属性，再叠加 ReactBits 气质**
- 希望引入 ReactBits `Magic Bento` 的排版思路来展示数据可视化模块
- `Magic Bento` 的借鉴重点为：
  - 卡片大小对比关系
  - 图表模块拼接方式
  - 主次信息层级与节奏感
- 工作台对 `Magic Bento` 的采用方式为：
  - **排版骨架 + 轻量 hover / 边框呼吸感**
  - 不直接照搬整套高饱和强特效风格
- `Magic Bento` 排布中的主卡为：
  - **近 7 天患者血糖均值趋势**
- `Magic Bento` 排布中的次级放大卡为：
  - **患者状态分布**
- 顶部四个指标卡保持为 **四个独立小卡**
- 指标区重点是样式精致化，而不是结构重组

#### 当前判断

- 工作台适合成为“结构升级”的重点模块，而不是“特效最重”的模块
- 强表现力已分配给：
  - 全局导航 `StaggeredMenu`
  - 登录页 `ProfileCard`
- 因此工作台更适合走：
  - 布局像 `Magic Bento`
  - 气质仍偏医疗后台
  - 动效控制在低干扰范围内
- 工作台最终目标不是做“炫技型数据大屏”，而是做“更有结构感的医疗工作台”

#### 默认限制

- 不默认引入强粒子特效
- 不默认引入高频 tilt / magnetism
- 不默认引入高饱和紫色辉光体系
- 图表阅读性优先于 hover 炫技感

#### 待继续确认

- 无（当前模块已通过）

### 7.3 患者管理

#### 已确认

- 本模块采取保守优化策略
- **不使用 ReactBits `CardSwap` 替代患者列表主视图**
- 患者列表主区域仍以高效扫描、筛选、分页、对比为优先
- 本模块优化重心转向：
  - **患者详情页**
- 患者详情页优化范围为左右两侧都调整
- 但优先顺序为：
  - **先优化左侧档案卡，再处理右侧主内容区**
- 左侧档案卡整体方向为：
  - **高级人物资料卡**
- 患者头像策略采用：
  - **患者姓名首字 / 姓氏生成头像**
- 患者状态作为最突出的身份信息
- 标签、糖尿病类型等信息作为次级信息排列
- 当前左侧档案卡方向已获通过
- 患者详情页右侧主内容区不再强调 tab 切换
- 改为 **纵向分段展开**
- 纵向展开顺序为：
  1. 随访
  2. 血糖
  3. 饮食
- 三个纵向区块统一采用：
  - **明显的大卡片区块**
  - 各自具备标题、摘要、内容层

#### 当前判断

- 患者列表属于高频检索型后台界面，不适合用强展示型轮换卡片承载主信息
- 患者详情页更适合作为“质感提升”和“信息层级重组”的重点区域
- 患者详情页将从“左资料卡 + 右侧 tab 容器”转向“左资料卡 + 右侧连续工作台”
- 血糖在患者详情页右侧区块中承担最强摘要职责

#### 默认限制

- 患者列表不做会影响扫描效率的结构性改造
- 不用轮换式、堆叠式、自动交换式卡片替代主列表
- 过滤、分页、删除、编辑跳转等列表功能保持原样

#### 待继续确认

- 无（当前模块已通过）

### 7.4 血糖模块

#### 已确认

- 血糖模块采用与工作台一致的 `Magic Bento` 优化思路
- 重点借用：
  - 排版骨架
  - 模块主次节奏
  - 轻量 hover / 边框呼吸感
- 不直接照搬高饱和强特效风格
- `Magic Bento` 排布中的主卡为：
  - **血糖趋势图**
- `Magic Bento` 排布中的次级放大卡为：
  - **分类分布**
- 异常统计 / 异常率摘要不单独拆成高权重卡片
- 异常统计 / 异常率摘要将：
  - **融入趋势图旁边的摘要区**
- “近期异常记录列表”继续保留
- 但其定位调整为：
  - **弱化后的辅助区**

#### 当前判断

- 血糖模块的核心判断入口应始终围绕趋势图展开
- 异常信息更适合成为趋势解读的一部分，而不是抢占独立主视觉
- 异常记录列表保留明细价值，但不应压过趋势与分布层级
- 血糖趋势图旁摘要区优先强调：
  - 达标率
  - 异常率
  - 记录总数

#### 默认限制

- 不将血糖模块做成大屏化数据展示
- 不使用高干扰粒子或强 tilt 动效
- 不让异常列表挤占第一视觉层级

#### 待继续确认

- 无（当前模块已通过）

### 7.6 饮食推荐

#### 已确认

- 本模块优化重心为：
  - **推荐审核详情**
- 审核详情适合引入 ReactBits `ScrollStack`
- `ScrollStack` 的用途为：
  - 组织推荐内容的分层阅读
  - 强化“先总览、后细看、再决策”的审阅节奏
- `ScrollStack` 不用于替代推荐列表主视图
- 审批操作区不应被滚动动效干扰
- 推荐审核详情允许采用：
  - **5 层及以上** 的分步审阅结构
- 当前已确认的审阅顺序为：
  1. 患者与推荐概览
  2. 餐次拆解
  3. 营养摘要
  4. 推送信息
  5. 审核决策

#### 当前判断

- 饮食推荐模块更适合做“内容审阅节奏”的升级，而不是列表结构重做
- `ScrollStack` 在该模块中的价值主要是帮助医生/营养师逐层审阅内容
- 审核通过 / 驳回等关键动作仍需保持清晰、稳定、低认知负担
- 审核决策区作为 `ScrollStack` 最后一层，但不能因此降低操作可达性

#### 默认限制

- 不用 `ScrollStack` 承载整个页面
- 不让审核按钮区随强动效漂移或隐藏
- 不牺牲当前审核流程的直达性

#### 待继续确认

- 无（当前模块已通过）

### 7.7 消息中心

#### 已确认

- 本模块优化重心为：
  - **消息详情抽屉**
- 消息详情抽屉整体方向为：
  - **折中型阅读面板**
- 即：
  - 整体保持克制
  - 但关键信息和操作区层次更强
- 详情抽屉顶部优先突出：
  - 消息类型
  - 已读 / 未读状态
- 详情抽屉的操作区采用前置策略：
  - 关联页面跳转按钮尽量顶部可见
  - 标记已读按钮尽量顶部可见
- 消息列表页本轮暂不做大结构重做

#### 当前判断

- 消息中心的核心问题不是“列表长什么样”，而是“打开一条消息后是否能快速判断并处理”
- 抽屉优化应提升扫读效率与处理效率，而不是增加展示型装饰

#### 默认限制

- 不重做消息分类逻辑
- 不重构消息列表主结构
- 不让消息详情抽屉因视觉强化而降低阅读清晰度

### 7.9 患者端 H5

#### 已确认

- H5 本轮进行：
  - **轻量优化**
- H5 整体气质偏：
  - **小工具**
- 关键词：
  - 轻
  - 快
  - 直接完成任务
- H5 两个页面都需要优化：
  - 血糖录入页
  - 通知查看页
- 优先级为：
  - **先优化血糖录入页**
- 血糖录入页优化重点同时包含：
  - 输入流程优化
  - 页面层级优化
- 但优先级为：
  - **先把输入流程做得更顺手**
- 通知查看页优化重点为：
  - 列表更清楚
  - 状态更明显

#### 当前判断

- H5 不追求重氛围或高展示感
- 应更像一个低负担、低学习成本的任务入口
- 血糖录入页优先解决“填得顺不顺手”
- 通知查看页优先解决“看得清不清楚”

#### 默认限制

- 不把 H5 做成重视觉展示页
- 不引入影响输入效率的强动效
- 不改动患者录入、通知查看的核心流程
- **不为 H5 单独引入新的 ReactBits 轻动效组件**

#### 待继续确认

- 无（当前模块已通过）

## 九、ReactBits 使用矩阵

| 组件 | 使用模块 | 使用目的 | 使用边界 |
|------|----------|----------|----------|
| `StaggeredMenu` | 全局框架层 | 替换固定侧栏，建立新的导航入口与展开节奏 | 不改菜单权限、路由与跳转逻辑 |
| `TextType` | 页面标题区 | 强化英文 kicker 与中文主标题的进入感 | 仅做一次性打字机式进入，不做高频轮播 |
| `ProfileCard` | 登录页 | 建立品牌化视觉主体 | 不改变登录表单、错误提示与登录跳转逻辑 |
| `Magic Bento` | 工作台、血糖模块 | 提供图表模块的排版骨架与主次层级 | 不直接照搬高饱和强特效体系 |

## 十三、当前系统结构对照与可行性说明

### 13.1 当前前端结构现状

基于当前代码结构，前端现状如下：

- 全局布局由 `frontend/src/components/layout/MainLayout.tsx` 承载
- 页面路由集中在 `frontend/src/router/AppRouter.tsx`
- 当前患者详情页仍为：
  - 左侧档案卡
  - 右侧 `Tabs` 容器
- 当前饮食推荐审核详情仍为：
  - `Drawer + Card` 组合
- 当前工作台、血糖模块已经具备独立图表组件，可作为后续 Bento 重排基础
- 当前 H5 仍为轻量表单 / 列表结构，未引入复杂动效依赖

### 13.2 当前 ReactBits 接入现状

目前项目中已存在的 ReactBits 封装仅包括：

- `AnimatedTitle`
- `EmptyMotion`

以下组件目前 **尚未接入当前代码库**：

- `StaggeredMenu`
- `TextType`
- `ProfileCard`
- `MagicBento`
- `ScrollStack`

因此后续实施这些模块时，属于：

- **有明确目标的新增组件接入**
- 而不是对现有同名组件进行简单替换

### 13.3 依赖与实施前置条件

根据当前 `frontend/package.json` 与 `h5/package.json`：

- `frontend` 当前未安装 `gsap`
- `frontend` 当前未安装 `lenis`
- `h5` 当前未安装额外动效依赖

这意味着：

- `StaggeredMenu`
- `TextType`
- `ProfileCard`
- `MagicBento`
- `ScrollStack`

如进入实施阶段，需先评估并补齐其依赖，再做组件落地。

### 13.4 与当前系统契合度判断

#### 高契合

- 全局框架层改造  
  当前导航集中在 `MainLayout`，具备统一替换入口

- 登录页替换为 `ProfileCard` 外壳  
  当前登录页结构简单，适合在不改登录逻辑的前提下重做视觉承载

- 工作台 / 血糖模块借用 `Magic Bento` 排版  
  当前图表已拆分为独立组件，适合重组布局层级

- 饮食推荐审核详情引入 `ScrollStack`  
  当前详情集中在抽屉内，天然适合重构为分层审阅结构

#### 中等契合（需要小规模结构调整）

- 患者详情页从 `Tabs` 转为纵向连续工作台  
  当前子模块已拆分为独立面板组件，可重组，但涉及页面结构调整

- 消息详情抽屉重做层级  
  当前抽屉结构稳定，适合做阅读层优化，但需谨慎处理顶部动作区

#### 明确不建议重构

- 患者列表主视图  
  当前为典型高频后台表格，适合保守优化，不适合展示型卡片轮换方案

- H5 引入新的重动效组件  
  当前 H5 结构与“小工具”方向一致，不宜引入额外动效负担

### 13.5 结论

当前方案与实际系统结构 **总体契合，可行性高**，但必须遵循以下执行方式：

1. 先接入缺失组件与依赖，再做页面级重构
2. 先改布局壳层，再改模块内部层级
3. 对患者详情页、饮食审核详情这类结构调整模块，必须模块级单独验收
4. 对患者列表、H5、系统管理等高频或轻量场景，坚持保守优化策略
