# 糖尿病随访系统 · 前端优化方案

> 版本：V0.2  
> 状态：讨论定稿，待按模块实施  
> 原则：不改变真实接口、不改变权限逻辑、不破坏既有功能

---

## 一、文档目的

本文件用于沉淀前端细节优化阶段的确认结论，确保后续实施严格围绕既有前后端结构展开，不因多轮讨论偏离目标。

---

## 二、本轮优化总目标

本轮前端优化聚焦三个方向：

1. 提升全局框架层的品牌识别度与进入感。
2. 在不破坏医疗后台可用性的前提下，引入 ReactBits 组件增强局部动效与排版。
3. 重组关键页面的信息层级，让医生端页面更符合“判断 - 处理 - 跟进”的阅读路径。

---

## 三、优化边界

### 3.1 可以调整的范围

- 页面布局层级
- 组件分组方式
- 标题区视觉与进入动效
- 卡片排版、间距、层次
- 空状态、过渡态、局部反馈
- ReactBits 组件的本地化封装与样式约束

### 3.2 明确不动的范围

- 前后端真实 API 对接方式
- 路由结构与权限判断
- 表单提交流程
- 业务字段语义与响应格式
- 登录、退出、跳转等核心行为
- H5 业务链路与后端契约

### 3.3 动效使用原则

- 动效集中在导航、标题、空状态、层级切换等低风险区域。
- 表格、表单、图表主体不使用高干扰动效。
- 所有增强动效必须可降级，不能影响读取和操作。

---

## 四、ReactBits 使用矩阵

| 组件 | 应用位置 | 用途 | 边界 |
| --- | --- | --- | --- |
| `StaggeredMenu` | 全局导航 | 替代固定侧栏，建立新的导航进入节奏 | 不改变菜单权限、路由、跳转 |
| `BlurText` / `TextType` | 顶栏标题区 | 增强页面标题进入感 | 仅用于标题，不扩散到正文 |
| `ProfileCard` | 登录页 | 作为登录页唯一主视觉外壳 | 不改登录逻辑 |
| `MagicBento` | 工作台、患者详情部分区块 | 重组信息层级与卡片节奏 | 仅借用结构，不做重特效 |
| `ScrollStack` | 饮食推荐详情、患者详情工作流区 | 构建分层阅读路径 | 不替代表格页主结构 |

## 五、模块结论总览

### 5.1 全局框架

- 侧边导航采用 `StaggeredMenu`。
- 顶栏标题统一保留，不再在页面内部重复放大标题。
- 标题文案动效采用 `BlurText` / 轻量化标题进入效果。

### 5.2 登录页

- 登录页使用 `ProfileCard` 作为唯一外壳。
- 不保留多层前景浮板。
- 表单需保持完整可见，不允许装饰层遮挡输入区。
- 卡片尺寸居中缩小，不占满大画布。

### 5.3 工作台

- 工作台采用 `MagicBento` 思路重组卡片布局。
- 放大主图表卡片，拉开卡片间距。
- 布局不占满整块画布，保留呼吸感。

### 5.4 患者管理

- 患者列表保持保守，不使用 `CardSwap`。
- 优化重点放在患者详情页。
- 左侧档案卡维持人物资料卡方向。
- 右侧改造为连续工作流视图。

### 5.5 血糖模块

- 血糖总览页延续 `MagicBento` 思路。
- 异常统计与异常率摘要并入趋势图相关摘要区。
- “近期异常记录列表”保留，但弱化为辅助区。

### 5.6 随访计划

- 本轮不做结构级视觉重构。

### 5.7 饮食推荐

- 优化重点为审核详情。
- 详情区可使用 `ScrollStack` 建立分层审阅节奏。
- 审核操作区必须始终稳定可达。

### 5.8 消息中心

- 重点优化消息详情视图与操作前置。
- 列表结构不做大改。

### 5.9 系统管理

- 本轮不做重构。

### 5.10 H5

- H5 仅做轻量优化。
- 保持“小工具”心智，不额外引入复杂 ReactBits 动效。

## 六、患者详情页专项定稿

### 6.1 总体方向

患者详情页右侧不再维持“随访 / 血糖 / 饮食”三大模块并列思路，而改为连续工作流视图，按医生真实阅读顺序推进。

### 6.2 工作流卡片顺序

右侧区固定拆分为 5 张卡：

1. 随访计划卡
2. 血糖趋势卡
3. 监测摘要卡
4. 近期记录卡
5. 饮食推荐卡

### 6.3 各卡片职责

#### 1. 随访计划卡

- 承接患者当前随访计划、执行状态、待办信息。
- 作为进入患者详情后的第一层背景判断。

#### 2. 血糖趋势卡

- 主体为近 14 日血糖趋势图。
- 右侧保留极简侧边摘要：
  - 最新值
  - 近 14 日异常点数
  - 趋势判断
- 不放完整分类均值和记录表格。

#### 3. 监测摘要卡

- 仅承接判断层指标，不放图表。
- 包含：
  - 30 日记录数
  - 异常率
  - 最近一条记录
  - 空腹均值
  - 餐后均值
  - 睡前均值
  - 随机均值

#### 4. 近期记录卡

- 这是右侧停留时间最长的明细操作层。
- 包含：
  - 近期记录表格
  - 分类筛选
  - 补录入口
  - 编辑 / 删除操作
- 编辑推荐使用抽屉或弹层，不采用行内长表单展开。

#### 5. 饮食推荐卡

- 采用时间线型结构。
- 按时间倒序展示：
  - 已生成推荐
  - 已提交审核
  - 审核通过 / 驳回
  - 已推送给患者
  - 患者已查看（若系统已有状态）

### 6.4 为什么采用拆卡方案

原问题并不是单纯的滚动参数错误，而是“趋势图、摘要、长表单”被绑定在同一张血糖卡中，导致 ScrollStack 的堆叠节奏始终与内容长度冲突。拆卡后：

- 趋势、摘要、明细三种阅读节奏被分离。
- `近期记录卡` 可自然成为长卡，不再和趋势图争空间。
- 饮食推荐卡作为末层接管，不会过早压住明细。

### 6.5 实施边界

可以调整：

- `PatientDetailPage` 右侧结构
- `PatientGlucosePanel` 的内部拆分与复用
- `PatientDietPanel` 的时间线展示方式
- ScrollStack 的层数、节奏与样式

不能调整：

- 左侧患者资料卡核心字段结构
- 后端接口
- H5 页面
- 其他模块的业务逻辑
- 路由与权限

## 七、与当前系统结构的契合判断

基于当前代码结构，本方案可行性高，原因如下：

- `PatientDetailPage.tsx` 已经是患者详情页主编排入口，适合承接右侧结构重组。
- 血糖、随访、饮食内容已拆为独立 feature 面板，可继续向下拆分为更细颗粒展示组件。
- 当前系统已完成真实接口联调，前端这轮主要是展示层重组，不依赖新增后端能力。

需要注意：

- 患者详情页的拆卡是结构性调整，必须在单页内逐步实施并人工验收。
- 不能为了适配动效而改写业务数据组织方式。

---

## 八、当前确认结果摘要

本轮前端优化已确认的重点模块为：

- 全局框架
- 登录页
- 工作台
- 患者详情页
- 血糖模块
- 饮食推荐
- 消息中心
- H5 轻量优化

当前最关键的新增结论为：

- 患者详情页右侧正式转为“工作流视图”
- 血糖区正式拆为“趋势卡 / 摘要卡 / 近期记录卡”
- 饮食推荐卡正式采用“时间线型”

后续实施必须以此为准，不再回到“三张大卡片硬堆叠”的旧方案。
