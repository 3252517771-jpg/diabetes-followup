## 三、模块边界与职责划分

### 3.1 架构分层

```
┌───────────────────────────────────────────────┐
│ routers/（路由层）                                │
│ 职责：参数提取 → 调用 service → 返回响应          │
│ 注意：薄层，不写业务逻辑，只做参数校验和响应包装     │
├───────────────────────────────────────────────┤
│ services/（业务逻辑层）                           │
│ 职责：核心业务处理、评分计算、趋势聚合、外部 API 调用│
│ 依赖：models/*, schemas/*                      │
│ 注意：不直接返回 Response 对象，只返回数据          │
├───────────────────────────────────────────────┤
│ models/（ORM 模型层）                            │
│ 职责：数据持久化、表关系定义、查询方法               │
│ 依赖：database.py                               │
├───────────────────────────────────────────────┤
│ schemas/（校验层）                                │
│ 职责：Pydantic 请求/响应校验，与前端类型对应        │
│ 依赖：无（纯类型定义）                              │
├───────────────────────────────────────────────┤
│ middlewares/utils/（基础设施层）                   │
│ 职责：认证注入、异常处理、日志、工具函数              │
│ 依赖：无                                         │
└───────────────────────────────────────────────┘
```

### 3.2 依赖规则

```
routers → services → models + schemas
routers → middlewares（认证注入）
services → models + utils
禁止：routers 直接调用 models
禁止：services 直接返回 Response 对象
禁止：schemas 出现业务逻辑
```

## 六、认证与权限体系

### 6.1 JWT 认证流程

```
注册 POST /api/auth/register
  → bcrypt 哈希密码
  → 写入 users 表
  → 返回成功

登录 POST /api/auth/login
  → 查 users 表
  → bcrypt 验证密码
  → 签发 JWT (payload: {sub: user_id, role: "doctor", exp: ...})
  → 返回 { access_token, token_type: "bearer", user: { id, name, role, department } }

后续请求
  → Header: Authorization: Bearer <token>
  → dependencies.get_current_user() 解码 JWT 并注入当前用户
```

### 6.2 JWT 配置

| 项目 | 值 |
|------|-----|
| 算法 | HS256 |
| 过期时间 | 1440 分钟（24 小时） |
| Secret | 环境变量 `SECRET_KEY`，64 位随机字符串 |

### 6.3 权限分级

| 级别 | 标签 | 可访问 | 说明 |
|------|------|--------|------|
| 0-公开 | Public | `/auth/login`, `/auth/register` | 无需 token |
| 1-登录 | User | 所有业务 API | 仅能操作自己权限范围内的数据 |
| 2-管理员 | Admin | `/system/*` | 系统管理 |

### 6.4 数据隔离规则

```python
# 医生：只能看到自己负责的患者
query = select(Patient).where(Patient.responsible_doctor_id == current_user.id)

# 科主任：看到全科患者
# 护士：看到所协助医生的患者
# 管理员：不操作患者数据
```

### 6.5 依赖注入实现

```python
# app/utils/dependencies.py
async def get_current_user(
    token: str = Depends(oauth2_scheme),
    db: AsyncSession = Depends(get_db),
) -> User:
    """验证 JWT，返回当前用户"""
    payload = decode_jwt(token)
    user = await db.get(User, payload["sub"])
    if not user or not user.is_active:
        raise HTTPException(401, "无效 token")
    return user

async def require_admin(current_user: User = Depends(get_current_user)) -> User:
    """管理员权限验证"""
    if current_user.role != 'admin':
        raise HTTPException(403, "需要管理员权限")
    return current_user
```

## 七、统一响应格式与异常处理

### 7.1 响应格式

```json
// 成功
{
  "code": 200,
  "message": "ok",
  "data": { ... }
}

// 失败
{
  "code": 400,
  "message": "字段 xxx: 具体原因",
  "data": null
}
```

### 7.2 HTTP 状态码使用规范

| 状态码 | 触发条件 | 消息模板 |
|--------|---------|---------|
| 200 | 成功 | "ok" |
| 400 | 参数校验失败 | "字段 xxx: 具体原因" |
| 401 | 未登录/token 过期 | "未授权，请重新登录" |
| 403 | 已登录但无权限 | "需要管理员权限" |
| 404 | 资源不存在 | "资源不存在" |
| 409 | 资源冲突（如用户名重复） | "用户名已存在" |
| 422 | 请求体格式错误 | "请求格式错误: 具体原因" |
| 500 | 服务器内部错误 | "服务器内部错误"（不暴露细节） |

### 7.3 异常处理器实现

```python
@app.exception_handler(HTTPException)
async def http_exception_handler(request, exc):
    return JSONResponse(
        status_code=exc.status_code,
        content={"code": exc.status_code, "message": exc.detail, "data": None},
    )

@app.exception_handler(Exception)
async def generic_handler(request, exc):
    logger.error(f"未捕获异常: {exc}")
    return JSONResponse(
        status_code=500,
        content={"code": 500, "message": "服务器内部错误", "data": None},
    )
```
