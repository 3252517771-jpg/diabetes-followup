## 三、模块边界与职责划分

### 3.1 架构分层

```
┌──────────────────────────────────────┐
│ Pages（页面层）                        │
│ 职责：路由节点，组合 features 组装页面   │
│ 依赖：features/*, components/layout/* │
│ 注意：页面不写业务逻辑，只做组合编排       │
├──────────────────────────────────────┤
│ Features（业务层）                      │
│ 职责：页面内可复用的业务模块              │
│ 依赖：components/*, hooks/*, services/*│
│ 注意：一个 feature 只做一件事             │
├──────────────────────────────────────┤
│ Components（UI 层）                    │
│ 职责：通用 UI 组件 + 布局组件 + 图表     │
│ 依赖：types/*, config/*, styles/*     │
│ 注意：纯展示，无业务逻辑，通过 props 驱动  │
├──────────────────────────────────────┤
│ Services / Store / Hooks（数据层）      │
│ 职责：API 请求、状态管理、自定义 Hook     │
│ 依赖：types/*, config/*               │
│ 注意：不和 UI 直接耦合，只暴露数据和操作函数 │
├──────────────────────────────────────┤
│ Utils / Config / Types（基础层）        │
│ 职责：工具函数、配置、类型定义            │
│ 依赖：无（纯函数/纯类型）                │
│ 注意：不允许依赖上层模块                  │
└──────────────────────────────────────┘
```

### 3.2 依赖规则（反向禁止）

```
Pages → Features → Components → 基础层
Pages → Features → Services/Hooks → 基础层
禁止：Components 依赖 Features
禁止：Services 依赖 Components
禁止：基础层依赖任何上层模块
```

## 六、核心接口协议

### 6.1 API 请求封装

```typescript
// frontend/src/services/request.ts
const BASE_URL = '/api'
const TOKEN_KEY = 'token'

export async function request<T>(
  endpoint: string,
  options?: RequestInit
): Promise<T> {
  const token = localStorage.getItem(TOKEN_KEY)
  const res = await fetch(`${BASE_URL}${endpoint}`, {
    ...options,
    headers: {
      'Content-Type': 'application/json',
      ...(token ? { Authorization: `Bearer ${token}` } : {}),
      ...options?.headers,
    },
  })

  // 统一错误处理
  if (res.status === 401) {
    localStorage.removeItem(TOKEN_KEY)
    window.location.href = '/login'
    throw new Error('未授权，请重新登录')
  }
  if (!res.ok) {
    const err = await res.json().catch(() => ({}))
    throw new Error(err.message || `请求失败 (${res.status})`)
  }
  return res.json()
}

// React Query 封装示例
export function useQueryWithError<T>(
  key: string[],
  endpoint: string
) {
  return useQuery<T>({
    queryKey: key,
    queryFn: () => request<T>(endpoint),
  })
}
```

## 八、认证与权限控制方案

### 8.1 JWT 认证流程

```
登录 POST /api/auth/login
  → 后端验证账号密码
  → 返回 { access_token, token_type, user_info }
  → 前端存入 localStorage
  → 后续请求通过 request.ts 自动注入 Authorization: Bearer <token>

401 响应
  → 清除 localStorage token
  → 跳转 /login
```

### 8.2 角色权限矩阵

| 页面/操作 | 系统管理员 | 科主任 | 医生 | 护士 | 营养师 |

| 用户管理 | ✓ | — | — | — | — |

### 8.3 路由守卫实现

```typescript
// 权限校验通过 RoleGuard 组件包裹受保护的路由
function RoleGuard({ roles, children }: { roles: RoleType[]; children: ReactNode }) {
  const { user } = useAuth()
  if (!user) return <Navigate to="/login" />
  if (!roles.includes(user.role)) return <Result status="403" title="无权限访问" />
  return <>{children}</>
}
```

## 十、组件封装规范

### 10.1 二次封装原则

- Ant Design 组件不直接写在页面代码中，统一在 `components/ui/` 中二次封装

## 十四、前后端接口验证流程

每个 Sprint 在开发完成后，须执行以下验证流程，确保前端接入的是真实后端 API 而非 Mock 数据：

### 14.1 验证规则

1. **后端先行**：每个模块对应 Sprint 中，后端应比前端提前 1-2 天完成 API 开发
2. **实时切换**：前端开发该模块时，request.ts 的 BASE_URL 指向真实后端，不再使用 Mock 数据文件
3. **逐接口验证**：每个 API 端点从前端页面触发真实请求，验证：
   - 请求是否正确发出（URL、Method、Headers）
   - 响应状态码是否符合预期（200/400/401/403 等）
   - 数据字段名和类型是否与 TypeScript 类型定义一致
   - 错误场景是否在前端正确处理（空白、加载态、错误提示）
4. **Mock 降级机制**：若后端 API 未按时就绪，允许临时切回 Mock，但需在 Sprint 验收前完成转换

### 14.2 Sprint 验收标准

| Sprint | 后端 API 基准 | 前端验收要求 |
|--------|-------------|------------|
| Sprint 1 | B1: 认证 API | 登录/注册流程接真实后端，不可用 Mock |
| Sprint 2 | B2: 患者 + 标签 API | 患者列表/详情/新建/编辑全部接真实 API |
| Sprint 3 | B2-B3: 血糖 + 计划 API | 血糖/计划模块接真实 API |
| Sprint 4 | B3: 计划执行 API | 计划看板数据来自后端 |
| Sprint 5 | B4: 饮食推荐 + 通知 API | 推荐/通知接真实 API |
| Sprint 6 | B5: 看板 + 系统管理 API | 看板/系统管理接真实 API |
| Sprint 7 | B6: 全量联调 | 全接口回归验证，零 Mock |

### 14.3 异常处理

若联调发现字段不一致或接口缺失，按以下优先级处理：
1. 后端调整 API 适配前端（优先）
2. 若后端调整成本过高，前端调整类型定义适配后端
3. 双方确认后更新设计文档中的 API 定义
