# 糖尿病随访系统 · 部署实施计划方案

> 版本：v1.0  
> 日期：2026-07-08  
> 适用范围：正式版单套环境部署，不包含开发环境、灰度环境、多活部署

---

## 一、目标与前提

本方案用于将糖尿病随访系统部署到现有 ECS 服务器，并确保**不影响服务器上已运行的其他项目**。

当前服务器已确认存在两套在线项目：

- `dji-rental`：`Nginx + Supervisor + Waitress`，占用 `127.0.0.1:5000`
- `xinxiaogan`：`Nginx + Supervisor + Uvicorn`，占用 `127.0.0.1:8001`

因此，本项目必须采用**独立目录、独立域名、独立本地端口、独立 Supervisor 进程、独立数据库文件**的并行部署方式。

---

## 二、部署结论

经讨论，正式版部署方案定为：

- 正式域名：`https://diabetes.moomlostdress.cn`
- H5 入口：与正式站点共用子域名，路径为 `https://diabetes.moomlostdress.cn/h5/...`
- 部署目录：`/opt/diabetes-followup/`
- 数据库方案：沿用服务器既有项目方案，使用 `SQLite`
- 运行方式：`Nginx + Supervisor + Python venv + 前端静态文件`
- 环境策略：**仅部署一套正式版环境**，不在服务器保留开发版或测试版

该方案的核心优势是：

1. 与服务器现有运维方式一致，维护成本最低
2. 不新增 MySQL/PostgreSQL 服务，适合当前轻量 ECS
3. 与现有项目完全隔离，风险最小

## 三、正式环境隔离边界

### 3.1 目录隔离

新项目独立部署在：

```text
/opt/diabetes-followup/
```

建议目录结构：

```text
/opt/diabetes-followup/
├── backend/                 # FastAPI 后端代码
├── frontend-dist/           # 医生端构建产物
├── h5-dist/                 # H5 构建产物
├── data/                    # SQLite 数据文件
├── logs/                    # 项目日志（可选）
└── .venv/                   # Python 虚拟环境
```

### 3.2 进程隔离

- 后端进程使用独立 `Supervisor` 配置
- 后端监听新的本地端口，例如 `127.0.0.1:8010`
- 禁止复用 `5000`、`8001`
- Nginx 为本项目新增独立 `server` 配置，不修改已有站点逻辑

### 3.3 数据隔离

正式版数据库文件独立存放，例如：

```text
/opt/diabetes-followup/data/diabetes_followup_prod.db
```

禁止与本地开发环境、历史 SQLite 文件或其他项目数据库共用数据文件。

### 3.4 配置隔离

正式版必须使用独立 `.env`：

```text
/opt/diabetes-followup/backend/.env
```

其中以下内容必须与本地开发环境隔离：

- `SECRET_KEY`
- `DATABASE_URL`
- `CORS_ORIGINS`
- `AI_API_KEY`
- `SERVER_CHAN_KEY`
- 推送开关与日志级别

## 四、推荐部署架构

```text
Internet
   ↓
Nginx : 80/443
   ├── /           → /opt/diabetes-followup/frontend-dist
   ├── /h5/...     → /opt/diabetes-followup/h5-dist
   └── /api/...    → http://127.0.0.1:8010/api/...
                           ↓
                     FastAPI + Uvicorn
                           ↓
                     SQLite (独立 .db 文件)
```

说明：

- 医生端与 H5 共用同一子域名，便于统一证书、统一回源和统一链接生成
- 静态资源由 Nginx 直接提供
- API 仅通过 Nginx 反代到本地后端端口
- 不开放 SQLite 到网络侧

## 五、H5 信息隔离策略

当前系统未接入实名校验、短信验证码或身份证认证，因此患者 H5 的信息隔离不能依赖“自然人身份识别”，而要依赖**受限访问令牌**。

正式版采用以下策略：

### 5.1 基础策略

- H5 链接内包含患者专属访问 `token`
- 后端仅根据 `token` 解析患者身份与访问范围
- H5 请求中即使传入 `patient_id`，后端也不得信任该参数，必须以 `token` 中的主体为准

### 5.2 正式版加固策略

为降低链接转发造成的越权风险，正式版增加以下约束：

1. **短期有效 token**
   - 推荐有效期：24 小时或 7 天
   - 过期后由医护端重新生成和推送

2. **作用域限制**
   - H5 token 仅允许访问患者端接口
   - 不得访问医生端、管理端或其他患者数据接口

3. **手机号后四位二次校验**
   - 患者打开链接后，先输入手机号后四位
   - 后端校验与档案手机号是否一致
   - 校验通过后才允许进入 H5 页面

4. **页面提示**
   - 明确标注“链接仅限本人使用，请勿转发”

该策略能在不引入重型认证体系的前提下，提升正式版的实际隔离强度。

## 六、部署实施步骤

### 阶段 1：部署前准备

- 确认域名 `diabetes.moomlostdress.cn` 的 DNS 已解析到当前 ECS
- 确认不与现有 `dji.moomlostdress.cn`、`xinxiaogan.moomlostdress.cn` 冲突
- 整理正式版 `.env` 参数
- 明确正式版初始管理员账号与密码

### 阶段 2：服务器目录与运行时准备

- 创建 `/opt/diabetes-followup/` 目录
- 创建 Python 虚拟环境
- 上传后端代码、医生端构建产物、H5 构建产物
- 创建 `data/` 目录和 SQLite 数据文件存放位置

### 阶段 3：后端部署

- 安装后端依赖
- 配置正式版 `.env`
- 执行数据库迁移或初始化建表
- 使用 `Supervisor` 托管 `uvicorn`
- 确认本地端口仅监听 `127.0.0.1`

### 阶段 4：Nginx 接入

- 新增 `diabetes.moomlostdress.cn` 独立站点配置
- `/api/` 转发到 `127.0.0.1:8010`
- `/` 指向医生端静态资源目录
- `/h5/` 指向 H5 静态资源目录或其路由入口
- 申请并配置独立 HTTPS 证书

### 阶段 5：正式版验证

- 验证首页、登录、患者管理、血糖、随访、饮食推荐、消息中心
- 验证 H5 链接生成、打开、录入、校验与过期行为
- 验证推送链路是否使用正式环境配置
- 验证正式库中用户、角色、患者数据与本地开发环境完全隔离

## 七、上线执行剧本

### 7.1 Git 分支策略

正式发布采用双分支策略：

- `master`：日常开发与验收主线
- `prod`：服务器正式运行分支

发布原则：

1. 开发与联调先在 `master` 完成
2. 本地验收通过后，再将 `master` 合并到 `prod`
3. 服务器正式目录固定跟踪 **Gitee 仓库的 `prod` 分支**
4. 禁止在服务器上直接修改业务代码并提交
5. 禁止将未验收完成的开发中间态直接推入 `prod`

### 7.2 正式目录策略

服务器正式目录采用“代码与运行时资产分离”方式：

- Git 工作副本：管理代码、依赖清单、构建脚本
- 非 Git 资产：`.env`、`data/`、`logs/`

即使执行 `git pull`，也不得覆盖以下内容：

- 正式环境配置
- 正式数据库文件
- 正式日志文件

### 7.3 标准发布顺序

正式发布按以下顺序执行：

1. 本地在 `master` 完成开发、自测、联调与人工验收
2. 将 `master` 合并到 `prod`
3. 推送 `prod` 到 Gitee
4. 登录服务器，进入 `/opt/diabetes-followup/`
5. 发布前手动备份 SQLite 正式库
6. 执行 `git pull`，拉取最新 `prod`
7. 安装或校验依赖
8. 重新构建医生端与 H5 静态资源
9. 重启 `Supervisor` 托管的后端服务
10. 进行线上冒烟检查
11. 冒烟通过后，发布完成

### 7.4 验收策略

正式发版采用“双阶段验收”：

#### 第一阶段：本地验收

在推送 `prod` 前完成：

- 前端构建通过
- H5 构建通过
- 后端可正常启动
- 核心真实 API 可调用
- 关键流程人工验收通过

#### 第二阶段：线上冒烟检查

服务器拉取并重启后，再执行一次正式环境快速检查：

- 首页可访问
- 登录可用
- 患者列表与详情可打开
- 关键图表加载正常
- H5 链接可打开
- H5 token 校验与手机号后四位校验正常
- 推送链路可用

该策略的目标是：既不把所有风险压到服务器，也不让服务器上线后完全失去验证环节。

## 八、上线验收清单

正式版上线前至少完成以下检查：

1. `diabetes.moomlostdress.cn` 可通过 HTTPS 正常访问
2. API 仅通过 Nginx 暴露，后端端口不对公网开放
3. `Supervisor` 能自动拉起后端进程
4. SQLite 文件位于独立目录，权限合理
5. 正式版 `.env` 未进入 Git 仓库
6. Server 酱推送使用正式配置，不复用测试环境数据
7. H5 token 为短期有效，并具备手机号后四位校验
8. 现有两个在线项目访问与进程状态不受影响

## 九、回滚策略

正式环境采用轻量回滚策略，不做蓝绿部署或双活切换。

### 9.1 回滚触发条件

若发布后出现以下情况，应执行回滚：

- 后端无法启动
- 登录或核心 API 异常
- 医生端页面白屏或关键模块不可用
- H5 无法打开或录入失败
- 推送异常影响正式业务

### 9.2 回滚方式

回滚采用：

1. 回退到上一个已知稳定的 `prod` commit
2. 重新构建前端与 H5
3. 重启后端服务

### 9.3 数据兜底

由于正式库使用 SQLite，且本阶段不配置自动备份，因此每次发布前必须：

- 手动备份生产 `.db` 文件

这样即便代码回滚不足以恢复问题，也仍保留当次上线前的数据副本。

## 十、风险与应对

### 8.1 内存较小

当前 ECS 内存仅 `1.6GiB`，不适合额外引入重量级数据库服务。  
应对：继续使用 SQLite，后端保持单进程部署。

### 8.2 H5 链接转发风险

当前系统不是强实名认证。  
应对：短期 token + 作用域限制 + 手机号后四位校验。

### 8.3 多项目共机风险

Nginx、Supervisor、证书、端口均为共享基础设施。  
应对：新增配置时必须采用“只增不改”策略，先体检、后接入、逐项验证。

---

## 十一、本阶段边界

本方案仅输出**正式版部署设计与实施计划**，不包含以下动作：

- 不直接修改 ECS 配置
- 不直接申请证书或执行 Nginx reload
- 不直接创建 Supervisor 配置
- 不直接上传代码或启动正式服务
- 不在本阶段执行真实上线

后续进入实施阶段时，必须按本方案逐项落地，并在每一步执行前再次核对服务器现状。

---

## 十二、推荐结论

对于当前这台已运行多项目、资源较轻的 ECS，最合适的正式版部署路线是：

**`diabetes.moomlostdress.cn` + `/opt/diabetes-followup/` + `SQLite` + `Nginx + Supervisor + Uvicorn` + H5 令牌隔离与手机号后四位校验。**

这套方案兼顾了：

- 与现有服务器环境的一致性
- 对已有项目的最小影响
- 正式版用户和权限数据的独立性
- 当前阶段可接受的安全性与运维复杂度

## 附录 A：服务器部署操作清单

本附录作为后续正式上线的执行剧本，采用“阶段步骤 + 关键命令示例”的写法。  
执行原则：**先检查、后变更；先备份、后发布；先本地验收、后线上拉取。**

### A.1 部署前核对

上线前先确认以下条件：

- 本地 `master` 已完成开发与人工验收
- 待发布代码已合并到 `prod`
- `prod` 已推送到 Gitee
- 域名 `diabetes.moomlostdress.cn` 已解析到 ECS
- 服务器现有项目运行正常

服务器侧检查示例：

```bash
supervisorctl status
ss -ltnp
nginx -t
```

重点确认：

- `5000`、`8001` 仍由旧项目占用
- 预定新端口（如 `8010`）未被占用
- Nginx 当前配置合法

### A.2 初始化正式目录

首次部署时创建独立目录：

```bash
mkdir -p /opt/diabetes-followup
cd /opt/diabetes-followup
mkdir -p data logs
```

若采用服务器直接拉取 Gitee：

```bash
git clone -b prod <GITEE_REPO_URL> /opt/diabetes-followup
```

若目录已存在，则只进入工作副本：

```bash
cd /opt/diabetes-followup
git branch
git remote -v
```

必须确认当前跟踪分支为 `prod`。

### A.3 Python 虚拟环境准备

正式版后端使用项目内独立虚拟环境：

```bash
cd /opt/diabetes-followup
python3 -m venv .venv
. /opt/diabetes-followup/.venv/bin/activate
```

安装依赖示例：

```bash
cd /opt/diabetes-followup/backend
/opt/diabetes-followup/.venv/bin/pip install -r requirements.txt
```

说明：

- 正式版不使用系统全局 Python 包作为运行依赖
- 所有后端命令统一通过 `/opt/diabetes-followup/.venv/bin/python` 或同目录下工具执行

### A.4 正式环境配置

正式版 `.env` 独立维护，不从仓库覆盖：

```bash
cd /opt/diabetes-followup/backend
cp .env.example .env
```

正式环境至少确认以下配置：

- `DEBUG=false`
- `DATABASE_URL=sqlite+aiosqlite:////opt/diabetes-followup/data/diabetes_followup_prod.db`
- `CORS_ORIGINS=https://diabetes.moomlostdress.cn`
- 独立 `SECRET_KEY`
- 正式版 `AI_API_KEY`
- 正式版 `SERVER_CHAN_KEY`

说明：

- `.env` 只在服务器本地维护
- 不允许通过 `git pull` 覆盖正式配置

### A.5 数据库初始化

正式库文件放置于：

```text
/opt/diabetes-followup/data/diabetes_followup_prod.db
```

若项目采用 Alembic：

```bash
cd /opt/diabetes-followup/backend
/opt/diabetes-followup/.venv/bin/alembic upgrade head
```

若依赖应用启动自动建表，也应在首次上线后额外确认数据库文件已生成且权限正确。

检查示例：

```bash
ls -lh /opt/diabetes-followup/data
```

### A.6 前端与 H5 构建

服务器拉取 `prod` 后，在正式目录内完成构建。

医生端构建示例：

```bash
cd /opt/diabetes-followup/frontend
npm install
npm run build
```

H5 构建示例：

```bash
cd /opt/diabetes-followup/h5
npm install
npm run build
```

构建完成后，需要确认最终静态资源目录与 Nginx 配置一致。  
若采用复制产物方式，需明确复制到：

- `/opt/diabetes-followup/frontend-dist`
- `/opt/diabetes-followup/h5-dist`

### A.7 Supervisor 配置

正式版后端由独立 `Supervisor` 项管理，建议命名：

```text
diabetes-followup
```

建议配置要点：

- `directory=/opt/diabetes-followup/backend`
- `command=/opt/diabetes-followup/.venv/bin/uvicorn app.main:app --host 127.0.0.1 --port 8010`
- `autostart=true`
- `autorestart=true`
- 独立 `stdout_logfile` / `stderr_logfile`

检查与重启示例：

```bash
supervisorctl reread
supervisorctl update
supervisorctl status
supervisorctl restart diabetes-followup
```

### A.8 Nginx 配置接入

Nginx 为正式版新增独立站点，不改现有两套项目的配置逻辑。

建议路由：

- `/` → 医生端静态资源
- `/h5/` → H5 静态资源
- `/api/` → `http://127.0.0.1:8010/api/`

基础检查示例：

```bash
nginx -t
systemctl reload nginx
```

若使用 Certbot 证书，证书申请与绑定应在域名解析完成后执行。

### A.9 标准发布流程

每次正式发版统一按以下顺序执行：

1. 本地在 `master` 完成开发、自测、联调与人工验收

3. 推送 `prod` 到 Gitee

### A.12 异常回滚操作

若发布后异常，立即执行轻量回滚：

1. 回退到上一个稳定 `prod` commit
2. 重新构建前端与 H5
3. 重启后端
4. 必要时恢复发布前备份的 SQLite 文件

Git 回退示例：

```bash
cd /opt/diabetes-followup
git log --oneline -5
git reset --hard <stable_commit>
```

说明：

- `git reset --hard` 属于正式环境高风险操作，执行前必须确认目标 commit 正确
- 若采用更保守方式，也可通过 `git checkout <stable_commit>` 临时验证后再恢复分支状态
