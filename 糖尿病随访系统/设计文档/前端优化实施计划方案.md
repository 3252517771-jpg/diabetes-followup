## 二、实施总原则

### 2.1 不变项

以下内容本轮默认不改：

- 前后端真实 API 契约
- 路由与权限逻辑
- 表单提交流程
- 认证、退出、角色控制
- H5 业务链路
- 后端数据结构与响应格式

### 2.2 可变项

允许调整：

- 页面布局壳层
- 标题区与导航呈现
- 卡片分组与信息层级
- 局部视觉动效
- 详情页内部阅读路径

### 2.3 验收原则

每个模块完成后必须验证：

1. 页面功能仍走真实接口
2. 原有操作入口未失效
3. 控制台无新增报错
4. 视觉变化与方案一致
5. 新动效未干扰核心操作

## 五、患者详情页专项实施说明

这是当前最关键、也最容易反复的模块，单独说明。

### 5.1 为什么要拆成五张卡

当前问题不是简单的滚动参数问题，而是：

- 趋势图属于快速判断内容
- 摘要指标属于压缩判断内容
- 近期记录表格属于长时停留内容

三者强行放进同一张血糖卡，会让堆叠节奏和内容节奏长期冲突。拆卡后，结构才会稳定。

### 5.2 实施方式

不新增新的业务接口，不改后端数据结构，而是基于现有面板继续拆分：

- 从 `PatientGlucosePanel` 中抽出：
  - 趋势展示块
  - 摘要指标块
  - 近期记录块
- `PatientDietPanel` 重排为时间线视图
- `FollowupPlanPanel` 继续作为首卡

### 5.3 明确边界

可以动：

- 右侧卡片数量
- 右侧滚动与堆叠节奏
- 血糖面板内部的 UI 拆分

不能动：

- 左侧患者资料卡核心结构
- 患者详情路由
- 后端接口字段
- H5 访问能力
