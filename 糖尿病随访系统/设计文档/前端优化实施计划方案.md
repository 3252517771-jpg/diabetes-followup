## 一、文档目标

本方案用于把 [前端优化方案.md](A:\医疗信息系统集成\糖尿病随访系统\设计文档\前端优化方案.md) 落实为**可执行、可验收、可回滚**的实施计划。  
所有任务均严格对照当前系统代码结构、现有真实接口和现有页面组织方式编写，避免盲改。

## 二、实施总原则

### 2.1 不变项

以下内容在本轮前端优化中默认不改：

- 前后端真实 API 对接方式
- 路由路径与权限逻辑
- 表单提交与业务流程
- 数据结构、字段语义、响应格式
- 认证、退出登录、跳转目标

### 2.2 可变项

以下内容允许小范围结构调整：

- 布局壳层
- 页面标题区
- 卡片分组与模块层级
- 抽屉内部内容组织
- 低风险动效与交互反馈

### 2.3 实施节奏

- 按模块逐个实施
- 每完成一个模块，单独构建、联调、人工验收
- 未验收通过，不进入下一个模块

## 六、模块级实施计划

### 6.1 模块 1：全局框架层

#### 目标

- 用 `StaggeredMenu` 替换固定侧栏
- 用 `TextType` 替换当前页面标题进入方式
- 保持菜单权限、路由、用户信息与退出逻辑不变

#### 当前对应文件

- `frontend/src/components/layout/MainLayout.tsx`
- `frontend/src/config/routes.ts`
- `frontend/src/index.css`

#### 计划新增文件

- `frontend/src/components/reactbits/StaggeredMenu.tsx`
- `frontend/src/components/reactbits/StaggeredMenu.css`
- `frontend/src/components/reactbits/TextType.tsx`
- `frontend/src/components/reactbits/TextType.css`

#### 实施任务

1. 抽离当前菜单配置，避免 `MainLayout.tsx` 内部硬编码继续膨胀
2. 在顶栏左上角植入 `StaggeredMenu` 触发器
3. 保留当前菜单项 key、权限判断与导航跳转映射
4. 将页面标题区改为 `TextType` 驱动
5. 为 `prefers-reduced-motion` 增加降级

#### 风险点

- `MainLayout` 当前承载导航与用户区，改动过大可能影响所有路由页
- `StaggeredMenu` 是高表现力组件，必须控制其作用域，不影响内容区点击与滚动

#### 验收点

- 所有页面跳转正常
- 管理员与非管理员菜单项显示正确
- 顶栏用户区与退出按钮行为不变
- 动效关闭场景仍可正常使用

### 6.2 模块 2：登录页

#### 目标

- 用 `ProfileCard` 建立新的登录视觉主体
- 保留用户名、密码、报错、登录按钮、默认测试账号提示

#### 当前对应文件

- `frontend/src/pages/LoginPage.tsx`
- `frontend/src/components/layout/AuthLayout.tsx`
- `frontend/src/index.css`

#### 计划新增文件

- `frontend/src/components/reactbits/ProfileCard.tsx`
- `frontend/src/components/reactbits/ProfileCard.css`

#### 实施任务

1. 接入 `ProfileCard` 组件并本地化 props
2. 将登录表单做成覆盖在卡片前景上的轻实体面板
3. 保留默认测试账号提示
4. 增加轻量背景英文短句/医疗感标语
5. 确保输入框聚焦、错误提示、按钮点击不受 tilt / glow 干扰

#### 风险点

- `ProfileCard` 原始视觉较强，容易压住表单可读性
- 表单浮层若层级处理不当，会影响输入与错误态阅读

#### 验收点

- `doctor01 / secret123`、`admin01 / secret123` 可正常登录
- 错误提示清晰
- 鼠标、键盘、触控操作正常

### 6.3 模块 3：工作台看板

#### 目标

- 借 `Magic Bento` 重排工作台信息层级
- 保持图表组件与真实 API 不变

#### 当前对应文件

- `frontend/src/pages/DashboardPage.tsx`
- `frontend/src/features/dashboard/DashboardTrendChart.tsx`
- `frontend/src/features/dashboard/DashboardStatusChart.tsx`
- `frontend/src/features/dashboard/DashboardPendingChart.tsx`

#### 计划新增文件

- `frontend/src/components/reactbits/MagicBento.tsx`
- `frontend/src/components/reactbits/MagicBento.css`
- 或针对本项目定制的：
  - `frontend/src/components/reactbits/MagicBentoGrid.tsx`

#### 实施任务

1. 保留顶部四指标卡为四个独立小卡
2. 以“近 7 天患者血糖均值趋势”作为主卡
3. 以“患者状态分布”作为次级放大卡
4. 为“血糖达标率”“待处理事项”增加轻量数字动态
5. 只借用 `Magic Bento` 的排版骨架与轻 hover 感，不引入强粒子效果

#### 风险点

- 若直接照搬原始 `Magic Bento` 视觉，会偏离医疗后台语气
- 图表容器尺寸变化可能影响 ECharts 自适应

#### 验收点

- 三组真实接口继续正常渲染
- 图表无白屏、无裁切
- 卡片主次层级明确

### 6.4 模块 4：患者管理（重点：患者详情页）

#### 目标

- 保持患者列表保守优化
- 重做患者详情页的视觉与结构层级

#### 当前对应文件

- `frontend/src/pages/PatientListPage.tsx`
- `frontend/src/pages/PatientDetailPage.tsx`
- `frontend/src/features/glucose/PatientGlucosePanel.tsx`
- `frontend/src/features/followup/FollowupPlanPanel.tsx`
- `frontend/src/features/diet/PatientDietPanel.tsx`

#### 实施任务

1. 不改患者列表主表格结构
2. 左侧档案卡改为高级人物资料卡气质
3. 用姓名首字 / 姓氏生成患者头像
4. 状态单独强化，标签与类型降为次级信息
5. 右侧从 `Tabs` 改为连续纵向工作台
6. 右侧顺序调整为：
   - 随访
   - 血糖
   - 饮食
7. 三块统一使用大卡片分段结构

#### 风险点

- 这是本轮最大结构调整之一，涉及 `PatientDetailPage.tsx` 主体布局重组
- 需确保子面板组件可直接从 `Tabs` 容器迁移到连续布局

#### 验收点

- 患者详情页加载、编辑跳转、H5 入口复制功能正常
- 右侧三块内容都能正常渲染
- 不出现布局错位与滚动冲突

### 6.5 模块 5：血糖模块

#### 目标

- 在工作台之外，把血糖总览与患者详情中的血糖区块统一到同一视觉语言

#### 当前对应文件

- `frontend/src/pages/GlucoseOverviewPage.tsx`
- `frontend/src/features/glucose/PatientGlucosePanel.tsx`
- `frontend/src/features/glucose/GlucoseTrendChart.tsx`
- `frontend/src/features/glucose/DailyRecordChart.tsx`

#### 实施任务

1. 以血糖趋势图为主卡
2. 以分类分布为次级放大卡
3. 将达标率、异常率、记录总数并入趋势图摘要区
4. 异常记录列表继续保留，但弱化为辅助区
5. 对患者详情血糖面板与血糖总览页建立统一的区块语言

#### 风险点

- `PatientGlucosePanel.tsx` 与 `GlucoseOverviewPage.tsx` 当前信息组织方式不同，统一时需避免逻辑混乱

#### 验收点

- 趋势图、统计、记录列表接口继续正常
- 异常信息的主次关系更清楚
- 不损失医生快速判断异常的效率

### 6.6 模块 6：饮食推荐

#### 目标

- 强化审核详情的“分层审阅”节奏

#### 当前对应文件

- `frontend/src/pages/DietManagePage.tsx`

#### 实施任务

1. 保留推荐列表主结构
2. 对详情 `Drawer` 内部做内容重组
3. 接入 `ScrollStack`
4. 将详情内容重组为 5 层以上审阅结构：
   - 患者与推荐概览
   - 餐次拆解
   - 营养摘要
   - 推送信息
   - 审核决策
5. 让审核决策作为最后一层，但保持按钮可达性

#### 风险点

- `ScrollStack` 依赖 `lenis`，需要评估其与当前抽屉滚动容器的兼容性
- 审核按钮区若处理不好，可能导致操作路径变长

#### 验收点

- 生成、通过、驳回、推送联动逻辑不变
- 详情抽屉信息层级更清楚
- 审核操作仍然顺手
