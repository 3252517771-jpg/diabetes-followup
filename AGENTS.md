## 二、执行铁律（按优先级排列）

### 原则1：架构 > 设计 > 样式

改代码前**必须确认**：
1. 是否符合目录结构（pages/ / eatures/ / components/ 分层，或 outers/ / services/ / models/ 分层）
2. 是否违反模块依赖规则（禁止反向依赖）
3. 再谈 UI 和样式

违反任意一条 → 退回重写。

### 原则2：真实后端接口优先，禁止默认 Mock

从 v1.0 起，前端页面和表单提交**必须真实对接 FastAPI 后端接口**，不得再以 Mock 数据作为默认运行路径。
pm run dev 必须走真实 /api 请求。

Mock 只允许作为显式调试模式存在：必须通过 VITE_USE_MOCK_API=true 或 
pm run dev:mock 手动开启。任何 Sprint 验收、功能联调、提交前自检，都必须基于真实后端接口返回结果完成，不能用 Mock 结果冒充完成。

### 原则3：一文件一职责

- 一个 .tsx 文件只导出一个组件
- 一个 .ts 文件只负责一个功能（config / types / utils 各自分开）
- 超过 300 行的组件必须拆解
- 一个 Python .py 文件只对应一个模块（model / router / service 各司其职）

### 原则4：不动现有设计文档

设计文档/ 下的五篇设计文档 + 前后端设计方案**禁止删除、移动、重命名**。实施过程中若发现与设计文档的矛盾，先记录问题，再与人工确认后修正。

### 原则5：数据库先行原则

每个模块开发时，后端必须先完成以下步骤，前端才能开始该模块的页面开发：
1. ORM 模型定义 + Alembic 迁移
2. Pydantic Schema 定义
3. Service 业务逻辑层
4. Router API 端点（可调用）
5. 前端接真实 API 开发页面

跳过后端直接在前端写页面 → 退回等待后端就绪。

### 原则6：版本管理纪律

**6.1 提交信息格式**：使用 gitmoji 作为前缀标签，格式为 <emoji> <中文或英文摘要>，摘要用祈使语气、不超过 72 字符。

| 场景 | gitmoji | 示例 |
|------|---------|------|
| 初始提交 | 🎉 | 🎉 初始化项目：糖尿病随访系统 |
| 新增功能 | ✨ | ✨ 添加患者列表 API |
| 前端页面/组件 | 🎨 | 🎨 实现患者列表页布局 |
| 修复 bug | 🐛 | 🐛 修复血糖趋势图日期越界 |
| 重构 | ♻️ | ♻️ 重构认证中间件 |
| 文档 | 📝 | 📝 更新 API 接口文档 |
| 配置/CI | 🔧 | 🔧 配置 CORS 允许前端域名 |
| 数据库迁移 | 🗃️ | 🗃️ 添加患者标签关联表 |
| 样式/UI | 💄 | 💄 调整 Dub 主题圆角体系 |

**6.2 提交前检查**：
- 不在提交中包含 .env、__pycache__、
ode_modules、*.db 等忽略文件
- 不提交包含 console.log、print() 调试代码的文件
- 不提交未处理边界情况的代码（见五）

**6.3 推送策略**：
- 每个里程碑完成且经人工确认后，执行 git add -A && git commit -m "..." && git push
- 禁止攒多个里程碑一次性推送
- 禁止 git push --force（除非人工明确要求）

**6.4 回滚**：
- 提交顺序错误或漏提 → git reset --soft HEAD~1 修正后重新提交
- 里程碑内容有误 → git revert <hash> 生成新的回滚 commit，不回退历史

## 三、项目边界（红线清单）

### ✅ 这是这个项目要做的（v1.0）

- 医生端 React SPA：患者管理 / 血糖数据 / 随访计划 / 饮食推荐 / 看板 / 系统管理
- 患者端 H5：血糖录入 / 饮食记录 / 通知查看
- 用户认证 + 角色权限（管理员 / 科主任 / 医生 / 护士 / 营养师）
- AI 饮食推荐（DeepSeek API 调用 + 营养师审核）
- Server酱推送 + APScheduler 定时任务
- ECharts 数据可视化（血糖趋势 / 达标率 / 状态分布）

### ❌ 这个项目明确不做

#### 全版本不允许

- HIS / EMR 医院系统对接
- 智能血糖仪等硬件接入
- 在线问诊 / 处方开药
- 文件附件管理
- 第三方登录 / 邮箱验证 / 社交功能
- 付费 / 会员体系

#### v1.0 不做（即使技术可行）

- 移动端 App / 响应式适配（医生端仅 1920×1080 桌面端）
- 微信小程序（仅 H5 页面嵌入微信）
- 自动解绑 / 过期检测微信 SendKey
- 推送数据统计 / 用户留存分析
- 数据大屏 / 可视化投屏
- 请求频率限制（留 Nginx 层）
- HTTPS 证书配置（部署时配置）

**任何 AI 建议做上述"不做"清单中的内容 → 直接否决，不予讨论。**

### 危险操作（必须提问确认）

1. **引入新前端框架/库**（超出 React / Ant Design / ECharts / TanStack Query 的技术栈）
2. **修改目录结构**（pages/ / eatures/ / components/ 三层划分是定死的）
3. **合并 Sprint 或调整 Sprint 顺序**（各 Sprint 的依赖关系是强制的）
4. **新增数据库表**（所有 12 张表已定稿，新增须经确认）
5. **跳过后端 API 直接在前端写静态数据展示**
6. **修改统一响应格式 {code, message, data}**（前后端契约是定死的）
7. **在 FastAPI 路由中直接操作数据库**（必须通过 services 层）
8. **修改 .gitignore**（可能导致敏感文件进入版本库）
9. **执行 git push --force**（覆盖远程历史，需人工明确要求）
10. **在一个 commit 中混合多个里程碑的内容**（必须按里程碑独立提交）

## 五、交付验证标准

每个 Sprint 完成后验证清单：

### 完成后自问 4 个问题

1. **能跑吗？** → 前端 
pm run dev 无报错，	sc --noEmit 通过；后端 uvicorn app.main:app 启动正常，/docs 可访问
2. **符合规范吗？** → 目录放对了吗？命名规范了吗？引用了正确的类型了吗？
3. **引用文档了吗？** → 是否参照 设计文档/ 和 前端/后端设计方案与实施计划.md 中的定义？不要凭记忆写代码。
4. **版本合规吗？** → .gitignore 覆盖完整了吗？有不该提交的文件吗？commit message 符合 gitmoji 规范吗？

### 验证成本控制

验收阶段只做**简单业务逻辑验证 + 人工验证**，禁止把验收扩展成长链路、全量链路、重复截图、反复抓包或多脚本交叉验真。

### 提交前 3 不提交

- ❌ 有 console.log 调试代码 → 删干净再提交
- ❌ 有 ny 类型 → 必须定义类型
- ❌ 有未处理的边界情况（空数据 / 加载中 / 错误 / API 不可用）→ 必须处理

## 七、阶段里程碑提交规则

### 7.1 规则（不可跳过）

每个明确的完成节点，必须走完以下流程才能进入下一个阶段：

`
阶段性任务完成
  → [代理人] 自检：npm run dev 无报错 + tsc --noEmit 通过 + 后端 /docs 可访问
  → [代理人] 准备提交清单（改了哪些文件 + 完成了什么 + 还有哪些已知问题）
  → [人] 确认：人工验收，认为可以提交
  → [代理人] 提交版本：git add -A → git commit -m "<gitmoji> 完成里程碑 M{N}" → git push
  → [代理人] 进入下一阶段
`

### 7.2 哪些节点必须触发此流程

里程碑采用**后端先行、前后端同步**策略。每个里程碑包含对应后端 Sprint 和前端 Sprint 的交付，以后端 API 可用为前置条件。

| # | 里程碑节点 | 后端交付 | 前端交付 | 验收标准 |
|---|----------|---------|---------|---------|
| M1 | 项目骨架 + 认证系统 | B1: FastAPI 项目初始化、数据库建表、认证 API | S1: 项目初始化、Ant Design 主题、布局框架、登录页 | 注册/登录全流程接真实后端 API |
| M2 | 患者管理模块 | B2前半: 患者 CRUD + 标签 API | S2: 患者列表/详情/新建/编辑页面 | 患者管理全流程接真实 API |
| M3 | 血糖数据模块 | B2后半: 血糖录入/趋势/统计 API | S4: 血糖录入/趋势图/统计页面 | 血糖模块接真实 API，ECharts 渲染正确 |
| M4 | 随访计划模块 | B3: 模板 CRUD + 计划执行 API | S3: 模板库/编辑器/计划看板 | 随访计划全链路接真实 API |
| M5 | 饮食推荐 + 通知模块 | B4: 推荐/审核 + 消息通知 API | S5: 推荐审核/消息中心页面 | 推荐审核流程 + 消息通知接真实 API |
| M6 | 看板 + 系统管理 | B5: 看板聚合 + 用户管理 + 配置 API | S6: 看板图表/用户管理/系统配置/H5 页面 | 看板/管理/H5 接真实 API |
| M7 | 全量联调 + 部署上线 | B6: 全接口回归 + 生产部署配置 | S7: 全接口联调 + 质量验收 | 全功能可用，零控制台报错 |

### 7.3 违反代价

- ❌ 跳过人工确认直接提交 → git reset --soft HEAD~1 回滚，重走流程
- ❌ 攒多个里程碑一起提交 → 逐 commit 回滚，逐个重新提交
- ❌ 提交信息不清（如 "fix bug" "update"）→ git commit --amend 重写 message
- ❌ 前端未接真实 API 即声称完成 → 退回等待后端接口就绪
- ❌ .env 或 *.db 被提交 → 立即从 git 追踪中移除并更新 .gitignore
